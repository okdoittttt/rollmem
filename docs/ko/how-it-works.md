# 동작 원리 {#how-it-works}

rollmem은 두 가지 상태를 유지합니다:

- **`buffer`** — 최근 턴들을 원문 그대로 담은 `list[Message]`.
- **`summary`** — 버퍼에서 밀려난 모든 내용의 누적 요약(`str`).

둘 다 `RollingMemory`와 `AsyncRollingMemory`의 공개 속성이므로 언제든 라이브
상태를 들여다보거나 직접 조정할 수 있습니다.

## 턴 하나의 생애주기

1. `add_message`(또는 role별 래퍼)가 턴을 버퍼에 추가합니다.
2. 추가할 때마다 rollmem은 버퍼의 총 토큰 수를 `max_tokens`와 비교합니다.
3. 버퍼가 예산을 초과하면 가장 오래된 턴들이 **퇴출(evict)**됩니다:
   `summarize_fn`을 통해 `summary`에 접혀 들어간 뒤 버퍼에서 제거됩니다.
   `summarize_fn`이 없으면 그냥 버려집니다.
4. `get_messages()`는 요약을 `system` 턴으로 앞에 붙인 버퍼를 반환하고,
   `get_context()`는 같은 내용을 프롬프트에 바로 쓸 수 있는 하나의 문자열로
   합칩니다.

## `max_tokens`의 의미

`max_tokens`는 **원문 최근 메시지 버퍼**의 예산입니다 — 누적 요약의 한도가
아니고, 모델의 생성 `max_tokens`(출력 한도)와도 무관합니다. 버퍼가 이 값을
초과하면 다시 예산 안에 들어올 때까지 가장 오래된 턴들이 요약으로 접혀
들어갑니다.

버퍼에는 항상 **최소한 가장 최근의 유닛**(아래 참고)이 남습니다. 그 유닛
하나가 예산을 초과하더라도, 가장 최근 턴이 요약으로 사라지는 일은 없습니다.

## Tool call 유닛 {#tool-call-units}

퇴출은 tool 상호작용의 일부만 떼어내는 일이 없습니다. 버퍼는 함께 퇴출되거나
함께 남는 구간인 **유닛(unit)** 으로 나뉩니다:

1. `tool_calls`를 가진 assistant 메시지는 자신부터 "그 호출들에 응답하는
   마지막 버퍼 메시지(`tool_call_id` 매칭)"까지가 한 유닛입니다 — 그 사이에
   끼어 있는 무관한 메시지도 포함합니다(이들을 남기고 호출만 퇴출하면 그
   뒤에 여전히 고아 결과가 남기 때문입니다).
2. 결과가 아직 버퍼에 없는 tool call 메시지는 그 자체로 하나의 유닛입니다.
3. 호출이 버퍼에 없는 tool 결과(예: 잘린 저장본에서 복원된 경우)는 메시지
   하나짜리 유닛으로, 평범하게 오래된 순서대로 처리됩니다 — 오류가 아닙니다.
4. 그 외 모든 메시지는 메시지 하나짜리 유닛입니다.

이로써 퇴출이 tool 상호작용을 쪼개는 일이 없습니다. tool call이 없는
대화에서는 모든 메시지가 각자 유닛입니다. 퇴출 후 버퍼가 무엇으로
*시작*할 수 있는지는 아래의 경계 정렬이 담당합니다.

## 정리(pruning) 알고리즘

정리는 의도적으로 보수적입니다:

1. 먼저 버퍼를 건드리지 않은 채, 버퍼가 예산에 들어오려면 가장 오래된 유닛
   중 몇 개를 내보내야 하는지 계산합니다.
2. 그 경계를 유효한 시작점에 정렬합니다: 살아남는 첫 메시지가
   **응답성(response-like)** 턴 — assistant 턴, tool 턴, 또는 고아 tool
   결과 — 이라면, 응답성이 아닌 첫 생존 메시지까지 유닛 단위로 퇴출을
   확장합니다. 그런 턴으로 시작하는 히스토리는 대부분의 프로바이더 API가
   거부합니다(예를 들어 Anthropic은 첫 메시지가 `user` role이어야 합니다).
   마지막 유닛 전까지 유효한 경계가 없으면 1단계의 경계를 그대로 둡니다 —
   더 퇴출해도 시작점을 고칠 수 없으므로, 대신 컨텍스트를 보존합니다.
3. 그다음 퇴출 대상 메시지 전부를 **단 한 번의** `summarize_fn` 호출로
   요약합니다.
4. 그 호출이 성공한 뒤에야 메시지들을 버퍼에서 제거합니다.

여기서 세 가지 실용적인 결과가 나옵니다:

- **정리 한 번당 요약 호출 한 번** — 메시지마다 호출하지 않으므로 여러 턴이
  한꺼번에 퇴출돼도 비용이 낮습니다. 정렬은 퇴출 묶음을 키울 수는 있어도
  호출 횟수를 늘리지는 않습니다.
- **퇴출 후 버퍼는 항상 유효한 턴으로 시작합니다.** assistant 메시지, tool
  메시지, 고아 tool 결과로 시작하는 일이 없습니다 — 남은 유닛 전부가
  응답성인 퇴화 케이스만 예외입니다(아래 한계 참고).
- **실패 안전성.** `summarize_fn`이 예외를 던지면 버퍼는 그대로입니다: 턴이
  유실되지 않고, 예외는 호출자에게 전파됩니다.

## 동기와 비동기 {#sync-and-async}

`AsyncRollingMemory`는 위의 모든 것 — 같은 유닛, 같은 보수적인 정리, 같은
직렬화 포맷 — 을 공유하며, `summarize_fn`으로 코루틴 함수를 받을 수
있습니다(일반 함수도 동작합니다). 비동기에만 해당하는 보장:

- **정리는 직렬화됩니다.** 내부 락이 정리를 한 번에 하나만 실행하므로, 동시
  태스크가 이중으로 퇴출하거나 서로의 요약을 덮어쓰는 일이 없습니다. 요약
  호출이 진행 중일 때 추가된 메시지는 버퍼 꼬리에 붙을 뿐, 절대 유실되지
  않습니다.
- **실패 안전성은 그대로입니다.** 요약기가 예외를 던지거나 태스크가 `await`
  중에 취소돼도 버퍼는 동기 클래스와 똑같이 무손상입니다.
- **`clear()`가 이깁니다.** 요약 호출이 진행 중일 때 `clear()`를 부르면,
  지워진 대화가 부활하는 대신 그 접기(fold)의 결과가 폐기됩니다.
- **태스크 안전, 스레드 비안전.** 인스턴스 하나는 단일 이벤트 루프 위의
  여러 asyncio 태스크 사이에서는 안전하지만, 스레드나 이벤트 루프를 넘나들며
  공유해서는 안 됩니다.

## 토큰 계산 {#token-counting}

모든 예산 판단은 `token_counter`를 거칩니다. 메시지 한 건의 텍스트를 받아
`int`를 반환하는 함수입니다. 전달되는 텍스트는 `Message.token_text()`입니다:
content에 각 tool call의 이름과 인자, `tool_call_id` 연결 정보를 더한 것으로,
tool 페이로드도 예산에 반영됩니다. tool 필드가 없는 메시지라면 정확히
`content` 그대로입니다.

기본 카운터는 단순한 단어 수 계산(`len(text.split())`)으로, 의존성 없이 바로
동작하도록 의도적으로 단순하게 만들었습니다. 실제 예산에는 `tiktoken` 같은
모델 정확도의 카운터를 주입하세요
([시작하기](getting-started.md#counting-tokens-accurately) 참고).

## 한계 {#limitations}

!!! warning "설계상 손실이 있습니다"
    오래된 턴은 반복적으로 요약에 접혀 들어가므로, 매 단계마다 세부 정보가
    흐려지거나 사라질 수 있습니다("전화 게임" 효과). 잃어서는 안 되는 내용이
    원문 버퍼에 남아 있도록 `max_tokens`를 충분히 크게 잡으세요.

- **요약 크기는 대신 제한해 주지 않습니다.** `max_tokens`는 원문 버퍼만
  제한합니다. rollmem은 현재 요약과 퇴출된 턴들을 `summarize_fn`에 넘기고,
  반환값을 그대로 저장합니다 — 요약을 압축된 상태로 유지하는 것은 콜백의
  몫입니다. 단순히 이어 붙이기만 하면 요약(그리고 `get_context()`)이 무한정
  커집니다.
- **경계 정렬은 최선 노력(best-effort)입니다.** 토큰 기준 경계 너머의 모든
  유닛이 응답성 턴으로 시작하면(예: assistant 메시지가 길게 이어지는 구간)
  확장은 일어나지 않고, 버퍼가 여전히 응답성 턴으로 시작할 수 있습니다.
  `system` 턴과 커스텀 role 문자열은 절대 응답성으로 취급되지 않으므로,
  버퍼가 대화 중간의 `system` 메시지로 시작할 수 있습니다 — 대부분의 API에는
  문제없지만, `system` 턴을 메시지 목록 밖으로 빼는 어댑터라면 직접 처리해야
  합니다.
- **복원된 버퍼는 재정렬되지 않습니다.** `from_dict`는 원문 그대로
  복원합니다: assistant 턴이나 고아 tool 결과로 시작하는 저장본은 다음
  퇴출이 일어날 때까지 그 상태로 유지됩니다([영속화](persistence.md) 참고).
- **카운터만큼만 정확합니다.** 기본 토큰 카운터는 대략적인 단어 수입니다.
  실제 예산에는 모델에 맞는 정확한 카운터를 주입하세요.
- **언어별 라벨을 붙이지 않습니다.** 요약은 라벨 없는 `system` 턴으로
  앞에 붙습니다. "지금까지의 대화:" 같은 라벨이 필요하면 직접 프롬프트를
  조립할 때 추가하세요.
- **기본적으로 인메모리입니다.** 상태는 메모리에 존재합니다.
  [`to_dict()` / `from_dict()`](persistence.md)로 저장하고 복원할 수
  있습니다.
